# B-CareHub ダッシュボード再設計 v3

**位置づけ**：v2 で構造化した情報レーンを前提に、**ダッシュボードを「今日の判断と対応をする場所」** へ純粋化する再設計。
**目的**：開いて 5 秒で「今日 何を・誰に対して・いつまでに する必要があるか」がわかる SaaS にする。

---

## 1. 現状の問題点の整理

| # | 問題 | 影響 |
|---|---|---|
| ① | 左上の職員プロファイル（番号・氏名・所属）が常時表示で領域を消費している | 情報密度の損失。サイドバーが太く見え、本来の業務メニューが視覚的に弱くなる |
| ② | 緊急アラートバーが大きすぎる（赤背景＋ボタン）。「目立たせる」と「整理する」が両立していない | 緊急 1 件でも視界の上 1/4 を占有し、他の情報が押し下げられる |
| ③ | 「影響イベント」「アラート」「タスク」「お知らせ」「活動ログ」「申し送り」が縦に積み上がっていて、目線が下方向に流れる構成 | どこから処理を始めればいいか分からない／同種情報の重複感 |
| ④ | 食事発注ステータスが「3 つのカードを並べただけ」で、業務上の主役という存在感が弱い | 食事は B-CareHub の主要ユースケース。ここで判断を完結できないと UI 価値が落ちる |
| ⑤ | サイドバーが長く、件数バッジがほぼ全項目に付いていて視覚ノイズになっている | 「どのバッジが本当に対応必要か」がパッと判別できない |
| ⑥ | カードがところどころ並んでいて、「管理画面の箱の積み重ね」に見える | SaaS としての整理感（情報のヒエラルキー）が弱く、テンプレ感が出る |

## 2. ダッシュボード再設計方針

### 2-1. ダッシュボードの定義を変える
- **v2 まで**：ダッシュボード ＝ 施設の状況サマリ（眺める場所）
- **v3 から**：ダッシュボード ＝ **今日の判断と対応をする場所**（動かす場所）

### 2-2. 情報を「上から処理する」レイアウトにする
ダッシュボードを 5 段で構成し、上から順番に処理するだけで朝の業務が終わる構造にする。

| 段 | 目的 | 主な要素 |
|---|---|---|
| **1 段目** | **今日の状況サマリ** | 横並び 5 つの数字（食事未確定／在庫不足／未確定請求／期限間近タスク／重要申し送り） |
| **2 段目** | **今日 判断・確定する** | 左：本日の食事発注ステータス（主役）／右：今日の要対応タスク |
| **3 段目** | **食事・請求に影響する変化** | 入院・外泊・一時帰宅・退院予定・退去予定の利用者リスト（1行に集約） |
| **4 段目** | **運用サマリー** | 横並びの軽い数字（カード化しない） |
| **5 段目** | **背景情報** | 在庫アラート／申し送り／活動ログを 3 カラムで小さく |

### 2-3. 情報レーンの整理（v2 から発展）
| 種別 | v3 での扱い |
|---|---|
| アラート | ダッシュボード 1 段目の数字 + 個別の発生箇所（食事ステータス・運用サマリーの中）にインライン化。**専用画面は廃止せず /inbox/alerts に残す** |
| タスク | ダッシュボード 2 段目 右側で主役の一つ |
| お知らせ | サイドバーから外し、**申し送り画面の上部にピン留め枠で吸収** |
| アクティビティ | ダッシュボード 5 段目右に折りたたみ |

### 2-4. UI ルール
- **大きい赤背景の警告ボックスは廃止**。重要度は左縦線 + 行ハイライト + 数字色で示す
- **カードは "上段の概観" と "下段の小ボックス" の 2 箇所だけ**。本文（食事ステータス・要対応タスク・影響イベント）は カード境界を最小化したオープンレイアウト
- **行リスト + 行クリックでドロワー** が基本。行末にボタンを並べない
- **数字フォントは tabular-nums、太字、ラベルは小さく**。SaaS らしい "数字の力" を出す
- **ステータスは「色 + テキスト + アイコン」**

## 4. 左サイドバー再設計案（詳細）

### 4-1. 構成
```
[ロゴ B-CareHub]
                          ← ここに「あすか苑」表示（小さく）

[運用]
  ダッシュボード
  利用者
  食事発注          3 ●
  月次請求         10 ●
  日用品            2 ●

[共有]
  申し送り          1 ●
  書類・タスク      4 ●
  アクティビティ

[管理]
  マスタ
  職員・権限
  監査ログ
```

### 4-2. バッジルール
| メニュー | バッジ条件 |
|---|---|
| 食事発注 | 当日の未確定 区分数（朝・昼・夕のうち） |
| 月次請求 | 当月の未確定件数 |
| 日用品 | 在庫不足件数（最低在庫を下回っている品目） |
| 申し送り | **未読の "重要" フラグ付きのみ** カウント。普通の申し送りはカウントしない |
| 書類・タスク | 期限超過 + 期限7日以内 の合計 |
| その他 | バッジなし |

色：
- **赤バッジ**：締切超過・在庫切迫など即対応
- **橙バッジ**：要対応だが当日中
- **灰バッジ**：参考

### 4-3. アクティブ表示
- 左に 3px の縦線（ブランド色）+ 背景 brand-50 + 文字 brand-700 + 太字

### 4-4. その他
- 「お知らせ」メニュー **廃止**（申し送り画面の上部に「お知らせ」セクションを設ける）
- 「アラート」メニュー **廃止**（ダッシュボード 1 段目に常時表示、専用画面は /inbox/alerts に残す）

## 5. 各セクションの表示項目

### 5-1. 今日の要対応バー（1 段目）

| 項目 | 数字の出し方 | クリック先 |
|---|---|---|
| **食事未確定** | 「2 区分」など、朝・昼・夕のうち未確定の区分数 | `/meals/2026-05-11` |
| **在庫不足** | 「2 品目」 | `/goods?filter=low` |
| **未確定請求** | 「10 件」 | `/billing?status=unconfirmed` |
| **期限間近タスク** | 「3 件」（期限超過 + 期限3日以内） | `/inbox/tasks?filter=urgent` |
| **重要申し送り** | 「1 件」（未読の重要のみ） | `/handovers?filter=important` |

- カード化しない。罫線で区切った横並び。
- 数字は太く大きく（22px）、ラベルは下に小さく（11px）
- 数字色：0 件は灰、それ以外は橙、緊急（食事未確定で締切間近など）は赤

### 5-2. 本日の食事発注ステータス（2 段目左・主役）

横並び 3 ブロック（朝・昼・夕）

```
┌─ 本日 5/11(月) の食事発注 ──────────────────────────────────────┐
│                                                                  │
│  朝食 ✓ 確定済              昼食 ⚠ 未確定          夕食 待機中    │
│  ──────────                 ──────────              ──────────  │
│  パン 18  ジュース 12        A社 14  B社 8           A社 12 B社 6│
│  確定 5/11 08:40            締切 10:30              締切 15:00   │
│                              残 1h 50m                            │
│                              未確定 3 名                          │
│                              前日比 −1                            │
│                                                                  │
│                              [詳細を見る][確定する]               │
└──────────────────────────────────────────────────────────────────┘
```

- 確定済ブロックは緑（薄い）、未確定で締切近いは橙（薄い）、待機中は灰
- ボタンは未確定ブロックにのみ表示

### 5-3. 今日の要対応タスク（2 段目右）

- 上部にセグメント切替：**[緊急] [本日中] [今週中]**
- 行リスト：1 行 1 タスク
  - 優先度（高 / 中 / 低 のラベル、色付き小さい）
  - タスク名
  - 対象利用者
  - 期限
  - 担当
  - 行クリックでドロワー（編集・完了・コメント）
- 「+ タスク追加」 リンク

### 5-4. 影響イベント（3 段目）

「食事・請求に影響する利用者ステータス変化」。1 行で集約。

| 列 | 内容 |
|---|---|
| 利用者 | 氏名・敬称 |
| 部屋 | 部屋番号 |
| 状態 | 入院中 / 外泊中 / 一時帰宅 / 退院予定 / 退去予定 |
| 期間 | 5/5〜5/13予定 |
| 影響 | 食事停止 ／ 日割対象 ／ 請求停止 等 |
| 対応 | [ステータス変更] |

ステータスバッジは色＋テキスト。退院予定や復帰漏れは橙〜赤で強調。

### 5-5. 運用サマリー（4 段目）

```
入居 10  ・  入院 1  ・  外泊 1  ・  一時帰宅 1  ・  空室 3  ・  今月請求予定 ¥6,432,100  ・  未確定請求 10
```

- 区切り点（・）で分けた 1 行サマリーバー
- 数字部分は太字、ラベルは弱文字
- カード化しない。罫線囲みのみ
- 数字クリックで対応画面（フィルタ付き）

### 5-6. 在庫アラート（5 段目 左）

```
[在庫アラート（2）]
おむつ Lサイズ    残 24 / 最低 40      要発注
使い捨て手袋 M    残 4  / 最低 10      切迫
[発注候補表へ →]
```

### 5-7. 申し送り（5 段目 中央）

```
[申し送り（直近3件）]
09:40  田中→鈴木様  退院日 5/13 確定...
09:15  鈴木→山本様  本日通所キャンセル
08:30  看護加藤→小林様  水分量管理（★重要）
[申し送り一覧へ →]
```

### 5-8. アクティビティ（5 段目 右）

```
[アクティビティ（過去24h・12件）]
▶ 展開して全件表示
```

折り畳みで占有面積を抑える。

## 6. UI コンポーネントの使い分け方針

| コンポーネント | 用途 | このダッシュボードでの使用 |
|---|---|---|
| **サマリーバー（罫線囲みの横並び）** | 一目の概観 | 1 段目（要対応バー）、4 段目（運用サマリー） |
| **オープンレイアウトのセクション**（カード境界を引かない） | 主役コンテンツ | 2 段目（食事ステータス・タスク）、3 段目（影響イベント） |
| **行リスト + 行クリックドロワー** | 詳細を確認・編集 | タスク、影響イベント、申し送り |
| **小カード** | 補助情報 | 5 段目（在庫・申し送り・アクティビティ） |
| **セグメント切替** | カテゴリ絞り込み | 要対応タスク（緊急/本日中/今週中） |
| **ステータスバッジ** | 確定／未確定／要対応／切迫 | 食事ステータス、影響イベント、タスク優先度 |
| **モーダル** | 破壊的操作の確認のみ | 発注確定／請求確定／ステータス変更 |
| **ドロワー（右からスライド）** | 詳細表示・編集 | タスク詳細、影響イベントの利用者詳細 |

**禁止**：
- 大きい赤背景の警告ボックス
- カードを 4 つ以上横並べ
- 影が深い派手なカード
- 1 行に 4 つ以上のボタンを並べる

- ≥ 1280px：完全な5段、2カラム
- 1024〜1279px：5段目を2カラム + アクティビティを下に

## 9. 実装時の注意（開発者へ）

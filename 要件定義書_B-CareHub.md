## 0. 本書の位置づけ

本書は、現在Googleスプレッドシート・紙・口頭・手作業で管理されている、住宅型有料老人ホーム本体および併設の訪問介護・訪問看護事業のバックオフィス業務を、Webシステムに一元化するための要件定義書である。

## 1. システム概要

### 1-1. システム名
**B-CareHub（ビーケアハブ）**

由来：Backlly が提供する Care 領域の業務 Hub（情報集約拠点）。利用者を中心に、食事・日用品・固定費・介護/看護利用料・書類・タスクをハブ的に一元管理することを示す。

### 1-2. システムの位置づけ
B-CareHub は、国保連請求やカイポケ・カナミックの「代替」ではなく、**住宅型有料老人ホームの現場と事務が日々使う「利用者中心の業務台帳」**である。

中心に置くのは「利用者ひとり」であり、その人にぶら下がる以下の情報を一元管理する。

- 入居・部屋・契約状況
- 食事（朝食パン・ジュース、昼食弁当、夕食弁当）
- 日用品（おむつ・パッド・消耗品）
- 固定費（家賃・共益費・水道光熱費・管理費）
- 介護利用料・訪問看護利用料・保険外サービス料
- 立替金・実費・その他個別請求
- 書類（保険証・受給者証・契約書類等）の有効期限
- 利用者ごとのタスク・申し送り
- 月次請求予定額（＝請求書を作成する前の段階で「いくらになりそうか」が見える状態）

### 1-3. システムの特徴
- **見える化優先**：数字だけでなく「その数字がなぜそうなったか」の内訳を必ず追える
- **現場と事務の両立**：食事発注は月間カレンダー、請求は表形式、で両者の慣れた見え方に寄せる
- **確定後ロック**：発注確定・請求確定後は、自動再計算で勝手に上書きしない
- **履歴の完全保存**：誰が・いつ・何を・なぜ変えたかを必ず残す
- **拡張前提**：MVPはシンプルに作り、freee連携・カイポケ連携・LINE通知等は後付けできる構造とする

## 2. 背景・現状課題

### 2-1. 現状の管理方法
- 利用者情報：Googleスプレッドシート + 紙の入居者台帳
- 食事発注：Googleスプレッドシートの月間カレンダー（業者別・日別の人数を手集計）
- キャンセル：LINEや口頭で現場から事務へ連絡 → 手書きで反映
- 日用品：紙の使用記録、または事務員の記憶
- 固定費・実費：エクセルで利用者別に都度集計
- 介護利用料：カイポケ等から打ち出した自己負担額を、別エクセルへ転記
- 請求書：エクセルから手入力でWord/PDF化

### 2-2. 具体的な課題

#### 食事発注の課題
1. 利用者ごとに「朝はパン、昼はA社弁当、夕はB社弁当」のように組合せが異なり、人別の食事設定が複雑
2. 昼と夕で弁当業者が違うため、業者別の集計が手作業
3. 単発キャンセル（明日のお昼だけ要らない）が日々発生
4. 「毎週木曜の昼は通所のためキャンセル」のような定期キャンセルがある
5. 入院・外泊・一時帰宅・退去で食事が止まるが、台帳と連動していない
6. 「土曜日に日曜分もまとめて配達」のような業者ごとの特殊配達ルールがある
7. 業者へ伝える発注数だけが残り、「その10食は誰の分か」がスプレッドシートから分からない
8. 月末の食費請求と、発注実績がそもそも紐づいていないので、突合に時間がかかる

#### 請求まわりの課題
1. 食事数と食費請求が連動していない（請求漏れ・過大請求が発生）
2. 日用品の利用者別使用数を集計するのに、紙の記録を月末にまとめて転記
3. 家賃・共益費・水道光熱費が利用者別の月額として一元管理されていない（部屋情報と紐づいていない）
4. 介護利用料・訪問看護利用料・保険外サービス費・立替金が別管理になっており、月次請求予定額が請求書を作るまで見えない
5. 「今月この利用者の請求はいくらになる予定か」を、月の途中で確認できない

#### 書類・タスクの課題
1. 介護保険証・受給者証の有効期限切れに気付くのが遅れる
2. 契約書・重要事項説明書の更新漏れ
3. 退去精算・家族連絡・ケアマネ連絡などの「やるべきこと」が個人のメモに散らばっている

### 2-3. 課題の本質
**「数字（発注数・請求額）はある」が「内訳（誰の・何が・なぜ）」が即座に追えない**ことが本質的な課題である。
本システムは、数字と内訳を常にセットで提示することを設計原則とする。

## 3. システム化の目的

### 3-1. ゴール
1. **利用者中心の業務台帳**を構築し、食事・日用品・固定費・介護利用料・立替金を1人の利用者にぶら下げて管理する
2. **食事発注を月間カレンダー形式**で可視化し、現場が紙の配食表と同じ感覚で扱える
3. **月次請求予定額**を、月の途中でも利用者別に確認できる
4. **発注確定・請求確定後の数字を保護**し、改ざん・上書き事故を防ぐ
5. **変更履歴**を全操作について保存し、トラブル時に誰がいつ何を変更したか追跡できる
6. **将来の外部連携**（freee, カイポケ, カナミック, LINE）に拡張できる構造にしておく

### 3-2. 業務上の効果
- 食事発注ミス（多発注・欠配）の削減
- 食費請求漏れの削減
- 月末締めの作業時間短縮（特に食費・日用品の集計）
- 書類期限切れの未然防止
- 利用者・家族・ケアマネへの説明責任に応えられる（履歴を即座に提示できる）

## 5. 想定ユーザー

### 5-1. ロール一覧

| ロール | 想定人物 | 主な利用シーン |
|---|---|---|
| 管理者 | 施設長、経営者 | 全機能、確定解除、マスタ、権限管理 |

| 閲覧専用 | 経営層、外部監査、家族（将来） | ダッシュボードと一部情報の閲覧 |

### 5-2. 利用デバイス
- 主：事務所のPC（Windows / Mac）
- 副：タブレット（現場のステーションに据え置き）
- スマホ最適化：Phase 1では優先度低（閲覧可能なレベルで足りる）

### 5-3. ユーザーの前提スキル
- スプレッドシートは使える
- ITスキルは中〜低（介護現場スタッフは特に低めを想定）
- 専門用語（介護保険・障害福祉用語）は理解している
- → UIは「業務用語に寄せる」「専門用語の標準化」を行い、IT用語を極力出さない

## 6. 全体業務フロー

### 6-1. 月次サイクル（標準業務カレンダー）

```
[月初]
 ├ 前月分の最終確定（食費・日用品・立替金）
 ├ 当月の固定費（家賃・共益費・水光熱費）を生成
 ├ 当月の食事カレンダーが自動生成される
 └ 書類期限・タスクの月初チェック

[月中（日次）]
 ├ 朝：当日の食事発注数を業者へ送る（カレンダー＋業者別発注表）
 ├ 随時：単発キャンセル・追加・手動調整
 ├ 随時：入院・外泊・退去のステータス更新 → 食事自動停止
 ├ 随時：日用品の使用登録
 └ 夕：翌日の発注数を確認（特に土曜は日曜分も含む業者あり）

[月末]
 ├ 食事発注の月末確定
 ├ 食費の自動集計
 ├ 日用品の月次集計
 ├ 介護利用料・訪問看護利用料の取込（カイポケ等のCSV、または手入力）
 ├ 立替金・保険外サービス料の最終入力
 └ 月次請求予定額の一覧確認 → 請求確定

[翌月初]
 └ 入金確認・未入金フォロー
```

### 6-2. 利用者ライフサイクル

```
[入居前] 見学・契約準備
   ↓ 入居予定登録（書類・部屋確保）
[入居] 入居日確定
   ↓ 食事設定・固定費設定・キーパーソン登録
[入居中] 日常運用
   ├ 入院 → ステータス変更 → 食事自動停止
   ├ 外泊 → ステータス変更 → 食事自動停止
   ├ 一時帰宅 → 期間キャンセル
   └ 通所等の定期外出 → 定期キャンセル
[退去] 退去日確定
   ↓ 退去精算（日割り、立替金清算）
[退去後] アーカイブ
```

## 8. 各画面の詳細要件

### 8-1. ログイン画面（Phase 1）
- メール + パスワード認証
- パスワードハッシュ化保存（bcrypt 等）
- ログイン失敗5回でロック（10分）
- 「ログイン状態を保持」チェックボックス
- パスワードリセット（メール送信）
- セッションタイムアウト：60分無操作で自動ログアウト

### 8-2. 施設全体ダッシュボード（Phase 1）

#### 上段：当日サマリ（数字を大きく）
- 現在の入居者数 / 定員 / 空室数
- 入院中人数 / 外泊中人数
- 本日の食事発注数（朝パン / 朝ジュース / 昼弁当A / 昼弁当B / 夕弁当A / 夕弁当B）
- 明日の食事発注数（同上、確定済みフラグ表示）

#### 中段：今月の財務サマリ
- 今月の請求予定額（合計）
- 利用者別請求未確定 ◯件
- 先月までの未入金額（Phase 2）

#### 下段：要対応リスト（クリックで該当画面へ遷移）
- 日用品の発注必要品目（最低在庫を下回ったもの）
- 未回収書類 ◯件
- 期限切れ・期限間近書類 ◯件（30日以内）
- 未完了タスク ◯件（期限超過は赤）
- 退去予定者（今月）
- 入居予定者（今月・来月）

#### 表示要件
- 各カードはクリック可能、対応する一覧画面へ遷移
- 数字フォントは見出し級（最低24px、できれば32px相当）
- カードに過度な影・グラデーションは付けず、罫線と余白で区切る
- 入院・外泊・期限切れなど「注意状態」は色（赤系・橙系）で示す

### 8-3. 利用者一覧画面（Phase 1）

#### 表示項目（テーブル）
| 列 | 内容 | ソート | 検索 |
|---|---|---|---|
| 部屋番号 | 101, 102, … | ◯ | ◯ |
| 氏名 | 漢字+フリガナ | ◯ | ◯ |
| ステータス | 入居中/入院中/外泊中/退去済 | ◯ | ◯ |
| 入居日 | YYYY-MM-DD | ◯ | |
| 食事設定 | 朝・昼・夕の有無を3アイコンで表示 | | |
| 今月請求予定額 | ¥nnn,nnn | ◯ | |
| 未回収書類 | バッジ数 | ◯ | |
| 未完了タスク | バッジ数 | ◯ | |
| 備考 | 1行省略 | | ◯ |
| 操作 | 「詳細」ボタン | | |

#### 絞り込み（左サイドペインまたは上部フィルタ）
- ステータス（複数選択）
- 部屋（部屋番号レンジ）
- 請求未確定のみ
- 未入金のみ
- 書類不足のみ
- タスクありのみ

#### 操作
- 新規利用者追加
- CSV出力（全列）
- 印刷（A4横、ヘッダー固定）

#### 表示要件
- 行高は詰めすぎず、44px程度（タブレットでも押しやすい）
- 部屋番号と氏名は固定列
- ヘッダーは固定（縦スクロールでも見える）
- ステータスは色付きバッジ（入居中：緑、入院：橙、外泊：黄、退去：灰）

### 8-4. 利用者詳細画面（Phase 1）

タブ構成（利用者1名のすべてを集約）：

```
[基本情報][食事][固定費・請求][日用品][書類][タスク][申し送り][履歴]
```

#### タブ1：基本情報
- 利用者ID（自動採番、表示のみ）
- 氏名、フリガナ、性別、生年月日、年齢（自動計算）
- 電話番号、住所
- 部屋番号（部屋マスタからプルダウン）
- 入居日、退去予定日、退去日
- ステータス（入居中/入院中/外泊中/一時帰宅/退去済）
- ステータス変更履歴ボタン
- 介護度、認定有効期間（開始・終了）
- 障害区分、受給者証期間
- 医療保険情報、負担割合
- キーパーソン情報（複数登録可）：氏名・続柄・電話・住所・請求書送付先フラグ
- 関係機関：担当ケアマネ、相談支援専門員、主治医、訪問診療先、訪問看護ST、訪問介護事業所、薬局、後見人
- 緊急連絡先（複数）
- 備考、申し送り（複数行・タイムスタンプ・記載者）

#### タブ2：食事
- 朝食設定：パンあり/なし、ジュースあり/なし、種類、開始日、終了日
- 昼食設定：業者（A/B/その他/なし）、食事形態（普通/きざみ/ミキサー等）、開始日、終了日
- 夕食設定：同上
- 設定変更履歴
- キャンセル登録（このタブから即時入力可能）
  - 単発キャンセル（日付＋食事区分＋理由）
  - 定期キャンセル（曜日＋食事区分＋開始日＋終了日＋理由）
  - 期間キャンセル（開始日＋終了日＋食事区分＋理由）
- 今月の食事実績サマリ（朝・昼・夕 ◯食ずつ）

#### タブ3：固定費・請求
- 家賃月額、共益費、水道光熱費、管理費、生活支援費、その他固定費
  - 各項目に：金額・請求開始日・請求終了日・日割りルール・請求先・支払方法
- 当月の請求予定額（明細）：固定費 / 食費 / 日用品 / 介護 / 看護 / 立替金 / その他
- 過去6ヶ月の請求推移（数字テーブル＋簡易折れ線、Phase 2）
- 入金状況（Phase 2）

#### タブ4：日用品
- 利用者別の日用品使用記録（日付・商品・数量・単価・備考）
- 当月使用合計（商品別）
- 当月請求対象金額
- 使用記録の追加ボタン

#### タブ5：書類
- 必要書類のチェックリスト形式
- 書類名・回収状況・有効期限・更新期限・ファイル添付・備考
- 期限切れ・期限間近は色で警告

#### タブ6：タスク
- この利用者に紐づくタスク一覧
- 期限・担当者・ステータス・優先度
- タスク追加

#### タブ7：申し送り
- 時系列の申し送り（記載日時・記載者・内容）
- スタッフ間の引き継ぎ用
- 新規記載・編集（編集は履歴に残す）

#### タブ8：履歴
- この利用者に関わるすべての変更履歴
- フィルタ：操作種別、操作者、期間
- CSV出力

#### 表示要件
- ヘッダーに利用者基本情報（氏名・部屋・ステータス）を常時表示（タブ切替時も）

### 8-5. 食事発注カレンダー画面（Phase 1：本システムの中核画面）

- 右サイドまたは下部：月合計テーブル
- 凡例：色・アイコンの意味

#### 各日付セル内の表示
```
 ┌──────────────┐
 │ 5/15 (土) [確定済]│
 │ 朝パン 18  ジュ 12 │
 │ 昼A 14  昼B  8    │
 │ 夕A 12  夕B  6    │
 │ ⚠キャンセル3 ✎調整2│

```

- 数字は明朝以外の太字数字フォントで見やすく

- 日曜：薄赤背景
- 土曜：薄青背景
- 祝日：薄ピンク背景（祝日マスタを保持）
- 平日：白

- 業者別単独表示（A社のみ、B社のみ）
- パン・ジュースのみ表示
- 印刷モード（白黒・罫線重視）

- 日付クリック → 日別食事詳細画面へ遷移
- 月合計の数字をクリック → 業者別発注表へ遷移
- 月締めボタン → 月内の未確定日を一括確定（管理者のみ）

#### 表示要件
- 表示は2秒以内（利用者100名×31日の負荷を想定）
- 印刷時はA3横で1ヶ月見開き、A4縦では半月ずつ
- 確定済み・未確定をはっきり区別
- スプレッドシートに慣れた人が「いつもの表より見やすい」と感じることを優先

## 9. 機能要件

### 9-1. 利用者管理機能（F-USER）
- F-USER-01：利用者の新規登録（必須項目チェック付き）
- F-USER-02：利用者情報の編集（履歴保存）
- F-USER-03：ステータス変更（入院・外泊・一時帰宅・退去）と、開始日・終了日の登録
- F-USER-04：ステータス変更時、対象期間の食事・固定費を自動停止／日割りに反映
- F-USER-05：キーパーソン・関係機関を複数登録可能
- F-USER-06：請求書送付先を利用者本人 / キーパーソン / 後見人から選択
- F-USER-07：CSV出力（利用者一覧）

### 9-2. 部屋・入居管理機能（F-ROOM）
- F-ROOM-01：部屋マスタ登録（部屋番号、定員、家賃ベース額、共益費ベース額）
- F-ROOM-02：部屋ごとの入居履歴（誰が・いつから・いつまで）
- F-ROOM-03：同一部屋に同時に複数人が入居している期間が発生しないようバリデーション

### 9-3. 食事設定機能（F-MEALCFG）
- F-MEALCFG-01：利用者ごとに朝・昼・夕の設定を登録（有無、業者、種類、形態、有効期間）
- F-MEALCFG-02：設定の変更履歴保存
- F-MEALCFG-03：将来日付からの設定変更（予約変更）

### 9-4. 食事キャンセル機能（F-CANCEL）
- F-CANCEL-01：単発キャンセル登録（日付・食事区分・理由）
- F-CANCEL-02：定期キャンセル登録（曜日・食事区分・期間・理由）
- F-CANCEL-03：期間キャンセル登録（開始日・終了日・食事区分・理由）
- F-CANCEL-04：入院・外泊・一時帰宅ステータス連動の自動キャンセル
- F-CANCEL-05：退去日以降の自動停止
- F-CANCEL-06：キャンセルの取消（履歴保存、理由必須）
- F-CANCEL-07：当日キャンセルの締切時刻バリデーション（業者ごとの締切後は警告）

### 9-5. 食事発注集計機能（F-ORDER）
- F-ORDER-01：日次の発注数自動集計（後述ロジック）
- F-ORDER-02：手動追加・手動減算（理由必須）
- F-ORDER-03：業者別配達ルール反映（土曜に日曜分含む等）
- F-ORDER-04：発注確定（日単位、月単位）
- F-ORDER-05：確定後は自動再計算で上書きしない
- F-ORDER-06：確定解除（管理者のみ、理由必須）

### 9-6. 月間カレンダー表示機能（F-CAL）
- F-CAL-01：月間カレンダー表示
- F-CAL-02：業者別フィルタ
- F-CAL-03：印刷・CSV
- F-CAL-04：月合計表示

### 9-7. 業者別発注機能（F-VENDOR）
- F-VENDOR-01：業者別発注表表示

## 12. 食事発注の集計ロジック

### 12-1. 基本式

```
最終発注数 = 基本対象者数 − キャンセル数 + 手動追加数 ± 手動調整
```

### 12-2. 計算手順（日付D・食事区分M・業者V について）

```
Step 1. D時点で在籍中の利用者を抽出
        条件: move_in_date ≤ D AND (move_out_date IS NULL OR D ≤ move_out_date)

Step 2. D時点で「在室中」のステータスを判定
        user_status_changes から、D を含む期間のステータスを取得
        除外: 入院中 / 外泊中 / 一時帰宅中 / 退去済

Step 3. 食事設定の有効レコードを抽出
        meal_settings WHERE user_id IN (Step1-2の結果)
                       AND meal_type = M
                       AND vendor_id = V（業者指定の場合）
                       AND valid_from ≤ D AND (valid_to IS NULL OR D ≤ valid_to)
        → 「基本対象者リスト」が決まる

Step 4. 定期キャンセルを反映
        meal_cancellations WHERE user_id IN (Step3 の対象者)
                              AND type = '定期'
                              AND weekday = D の曜日
                              AND period_start ≤ D
                              AND (period_end IS NULL OR D ≤ period_end)
                              AND meal_type = M
        → 該当者を対象から除外

Step 5. 期間キャンセルを反映
        meal_cancellations WHERE type = '期間'
                              AND period_start ≤ D ≤ period_end
                              AND meal_type = M
        → 該当者を対象から除外

Step 6. 単発キャンセルを反映
        meal_cancellations WHERE type = '単発'
                              AND target_date = D
                              AND meal_type = M
        → 該当者を対象から除外

Step 7. ステータス連動キャンセル
        Step 2 で除外済みのため、ここではStep 2の漏れがないかチェック

Step 8. 手動追加・手動調整を反映
        meal_adjustments を集計
        → 数値を加減算

Step 9. 業者別配達ルールを反映
        当該業者の delivery_rule_json を評価：
        - saturday_includes_sunday = true なら、D=土曜のとき翌日分も加算
        - no_sunday_delivery = true なら、D=日曜は 0 を返す（前日に積まれる）
        - holiday_before_includes_holiday = true なら、祝日前日に祝日分を加算
        - no_delivery_weekdays に該当する曜日は 0

Step 10. 確定済みフラグの確認
         meal_orders に confirmed=true のレコードが既に存在する場合、
         Step 1-9 の再計算結果ではなく、保存値を返す
         （= 確定後の上書き禁止）
```

### 12-3. 配達ルール例

| 業者タイプ | ルール例 |
|---|---|
| 弁当会社A | 月〜土配達、日曜分は土曜に積む |
| 弁当会社B | 月〜日配達、特殊ルールなし |
| パン業者 | 月・水・金のみ、複数日分まとめ配達 |
| ジュース業者 | 月初に1ヶ月分配達（数量管理のみ） |

### 12-4. 計算タイミング

| タイミング | 計算範囲 | 確定済との関係 |
|---|---|---|
| 利用者の食事設定変更時 | 翌日以降の未確定日 | 確定済は対象外 |
| キャンセル登録時 | 該当日（未確定） | 確定済は対象外 |
| ステータス変更時 | 期間内の未確定日 | 確定済は対象外 |
| カレンダー画面表示時 | 表示月の未確定日 | キャッシュ可 |
| 業者別発注表表示時 | 表示月の未確定日 | キャッシュ可 |
| 月締めバッチ（任意） | 当月の未確定日 | 確定済を保存 |

### 12-5. 例外パターン
- **当日キャンセル**：業者の締切時刻を過ぎている場合、UI上で警告 + `billable=true` のフラグを既定でON
- **遡及変更**：過去日付の食事設定変更は、確定済の発注を変更しない（請求側で別途調整）
- **業者変更**：利用者の業者を月途中で変更する場合、新旧両方の `meal_settings` を期間で区切る

## 13. 請求管理の集計ロジック

### 13-1. 月次請求予定額の集計式

利用者U、年月Y/M の請求予定額：

```
合計 = 固定費合計 + 食費 + 日用品費 + 介護自己負担 + 看護自己負担
       + 保険外サービス費 + 立替金 + その他費用
```

### 13-2. 固定費の集計

各 user_fixed_fees レコードについて：

```
入居月の請求 = (proration_rule に基づき計算)
中間月の請求 = monthly_amount
退去月の請求 = (proration_rule に基づき計算)
```

**日割り計算式（例）：**
- proration_rule = "日割りあり"：`monthly_amount × 当月の在籍日数 ÷ 当月の日数`
- proration_rule = "日割りなし"：`monthly_amount`
- proration_rule = "入居月無料"：入居月のみ 0、それ以降は満額
- proration_rule = "退去月満額"：退去月も `monthly_amount` 全額

### 13-3. 食費の集計

```
食費 = Σ(M食事区分別) ( 当月の利用者U の対象食数 × 食費単価 )

対象食数 = (基本対象日数 − キャンセル日数 [billable=falseのみ])
```

- `billable=true` のキャンセル（業者締切後の利用者都合キャンセル等）は **請求対象に含める**
- 単価マスタは `billing_items.default_unit_price` を月初時点でスナップショット

### 13-4. 日用品費の集計

```
日用品費 = Σ daily_goods_usage WHERE user_id=U
            AND used_date IN [Y/M]
            AND billable=true
        の amount
```

### 13-5. 介護・看護利用料

- 月次でCSV取込 or 手入力で `monthly_billing_details` に直接登録
- Phase 1 は手入力。Phase 2 でCSV取込

### 13-6. 立替金・その他費用

- 都度登録された `monthly_billing_details` のうち、対象月分を集計

### 13-7. 請求確定後の挙動

- 月次請求確定 = `monthly_billings.status = '確定済'` 以降
- 食事キャンセル・日用品使用記録・固定費契約の変更があっても、確定済の請求は再計算しない
- 確定解除（管理者のみ、理由必須）した場合のみ、再計算される

### 13-8. 請求書送付先の決定

利用者の `billing_recipient_type` に従い、対応する contact を取得。
不在の場合は本人住所へ送付（警告マークを表示）。

## 14. 日用品管理の運用ロジック

### 14-1. 商品の二系統管理

| 系統 | 例 | 在庫 | 利用者請求 |
|---|---|---|---|
| 施設共通消耗品 | 共用トイレットペーパー、共用洗剤 | 在庫管理する | 利用者請求しない |
| 利用者請求対象品 | おむつ、パッド、個別シャンプー | 在庫管理する | 利用者請求する |

### 14-2. 利用者別使用記録の入力フロー

1. 現場スタッフが利用者詳細画面のタブ4「日用品」、または日用品管理画面のサブタブ「利用者別使用」から入力
2. 商品プルダウン → 数量 → 日付（デフォルト今日） → 保存
3. 単価は商品マスタからスナップショット（後でマスタ単価が変わっても影響しない）
4. 月次請求時に当月分が自動集計される

### 14-3. 在庫の補充フロー

1. ダッシュボード or 日用品管理画面の「発注候補」を確認
2. 発注先別にグルーピングされた一覧を印刷 or CSV出力
3. 業者へ発注（システム外）

## 16. 非機能要件

### 16-1. セキュリティ
- HTTPS必須
- ログイン必須、外部公開URLでの認証なし閲覧禁止
- すべての操作ログ（audit_logs）に IPアドレス、操作者を記録

- セッションハイジャック対策（CSRF トークン、SameSite Cookie）
- SQLインジェクション・XSS対策（ORM経由、テンプレート自動エスケープ）
- アップロードファイルのウイルススキャン（Phase 2）
- 個人情報保護法・介護事業者ガイドラインに準拠

### 16-2. パフォーマンス
- 月間カレンダー（利用者100名・31日・業者5）：初回表示 2秒以内
- 利用者一覧（100名）：1秒以内
- 月次請求一覧（100名）：2秒以内

- DB：PostgreSQL を想定、適切なインデックス、N+1 を作らない

### 16-3. 可用性
- 稼働率：99%（月7時間程度のダウンタイム許容）

### 16-4. 保守性・拡張性

### 16-5. バックアップ・復旧
- DB日次バックアップ（最低30日保持）

### 16-7. ログ

## 20. 開発時の注意点

### 20-1. ドメイン理解の注意

- **「入居中」と「在室中」は違う**：入居中だが入院中の人がいる。食事は「在室中かつ食事設定あり」で初めて発注対象になる
- **「キャンセル＝請求しない」ではない**：業者締切後のキャンセルは請求対象になることがある。`billable` フラグを必ず持つ
- **「退去日」の解釈**：退去日当日に食事を出すか出さないかは施設による。退去日を含めるか含めないかを設定可能にする
- **「家賃」の解釈**：日割りルールが施設・利用者契約ごとに異なる。一律ロジックにしない
- **「ケアマネ」と「相談支援専門員」は別**：介護保険利用者と障害福祉サービス利用者で異なる
- **「介護度」「障害区分」は両方持つ利用者がいる**：両方の認定期間を別管理する

### 20-2. 例外対応の重視

介護現場は例外だらけである。以下を必ず実装すること。

- **すべての主要画面に備考欄**：利用者・食事・キャンセル・固定費・日用品・請求の各レコードに必ず備考欄を用意
- **手動調整の理由必須**：数値の自動計算を上書きする操作は理由を必須にする
- **変更履歴は「なぜ」も残す**：いつ・誰が・何を、だけでなく「なぜ」を残す
- **確定後の上書き防止**：誤って確定済みを変更しないよう、UI上のロック表示と、サーバー側の二重チェック
- **利用者別の内訳確認**：どんな集計画面からも、その数字の元となった利用者リストにドリルダウンできる

### 20-3. UI設計の注意（M-HAND等のデザイン哲学を踏まえて）

- **「いつもの表より見やすい」を目標にする**：スプレッドシートからの移行なので、まず情報密度を落とさない
- **過剰な装飾を避ける**：意味のないグラデーション、過剰なシャドウ、生成AI風の無駄なカードUIを使わない
- **配色は限定的に**：ベースは白・薄灰・濃灰、注意は橙、警告は赤、確定済は緑、リンクは青、の5色程度
- **数字は大きく、太く、右寄せ**：発注数・請求額は最も読みやすい状態にする
- **罫線と固定ヘッダー・固定列を活用**：エクセル感覚で使える
- **印刷を最初から考慮**：画面表示と印刷で破綻しないCSS（@media print）を必ず書く
- **アイコンは意味が一目で分かるものだけ**：抽象的なアイコンより、ラベル併記
- **タブで情報を整理**：利用者詳細は情報量が多いため、タブで分割（基本/食事/請求/日用品/書類/タスク/申し送り/履歴）

### 20-4. データ移行の注意

- 現行スプレッドシートからの移行は、CSVインポート用の管理者画面を用意
- 移行前に「利用者ID命名規則」「部屋番号」「業者マスタ」「キャンセル理由」を整理しておく
- 過去の食事発注実績の移行は MVP では対象外（当月以降から開始）
- 利用者基本情報・固定費契約・食事設定は MVP リリース前に登録完了する想定

### 20-5. テストデータ・運用テスト

- ステージング環境で「ダミー利用者20名 × 1ヶ月」を流して、月次サイクル全体を通しで検証
- 特に：入院・外泊・退去・新規入居が月内に混在するケース
- 業者の配達ルール（土曜に日曜分含む、祝日前まとめ等）を必ず実データで検証
- 月締め当日に「請求確定 → 解除 → 再確定」の動線を必ず通す

### 20-6. 法令・ガイドライン

- 介護保険法、老人福祉法、有料老人ホーム設置運営標準指導指針
- 個人情報保護法、医療・介護関係事業者における個人情報の適切な取扱いのためのガイダンス
- 電子帳簿保存法（請求書PDF保存は Phase 2 以降で対応）
- インボイス制度（請求書フォーマットは Phase 2 で適格請求書対応）

### 20-7. ローンチ後の運用支援

- 月次の不具合報告・改善要望を受け付ける窓口（事務担当者からのフィードバック）
- 操作マニュアル（ロール別、画面別）
- スタッフ向け勉強会（リリース時、半年後、新人加入時）
- 月次の利用状況レポート（操作ログから、未使用機能・困っている画面を特定）

## 付録 C：受け入れテスト用シナリオ例

### シナリオ1：通常の入居者の月次サイクル
- 利用者X（5/1入居、朝パン+ジュース、昼A、夕B）
- 5/15に1回だけ昼食キャンセル
- 5/20-22 入院 → 食事自動停止
- 5/31 月末確定
- 期待値：朝食 (31-3) = 28食、昼食 (31-1-3) = 27食、夕食 (31-3) = 28食、食費 = それぞれの単価×食数の合計

### シナリオ2：月途中入居
- 利用者Y（5/15入居、家賃 ¥80,000、日割りあり、5月の日数31日）
- 5月の家賃：¥80,000 × 17日 ÷ 31日 = ¥43,871（端数処理は切り捨て）

### シナリオ3：業者の特殊配達ルール
- 弁当会社A（土曜に日曜分含む）
- 利用者10名が昼食A、5/1〜5/31
- 5/3（土）：通常の土曜10食 + 翌日分10食 = 20食
- 5/4（日）：0食（配達なし）

### シナリオ4：手動調整
- 5/10（火）の夕食B、自動計算では8食
- 急遽親族訪問で2名分追加 → 手動追加2、理由「来客」
- 最終発注数 = 10食、履歴に「来客」が残る

### シナリオ5：確定後のロック
- 5/10 の発注を確定済みに
- 確定後に利用者Zが5/10の昼食をキャンセル登録 → 警告表示、発注数は変わらない
- 請求側で5/10の昼食を請求対象から外すかを別途判断

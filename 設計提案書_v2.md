## 1. 現状診断

v1 プロトタイプで観測された情報構造の問題。

### 1-1. 「タスク」「通知」「お知らせ」「KPI」が混在
- **タスク**：自分が動かないと進まないアクション要件
- **通知**：他者の操作結果・システム自動処理の結果報告
- **お知らせ**：施設管理者からのアナウンス（業務ルール変更等）
- **KPI**：数字のサマリ

v1 ではダッシュボードで「タスク」と「通知」の境界が曖昧で、両方に「確認 →」ボタンが付いていた。**タスクは押すと作業画面、通知は押すと既読化、お知らせは押すと詳細表示**という、押した後の挙動が違うはずなのに同じボタンになっている。

### 1-2. 「入居・財務サマリ」と「月次請求」の関係が冗長
- 入居数、空室、入院数 → 入居サマリで完結
- 今月の請求予定額、未確定請求 → 月次請求画面の冒頭に置くべき
- ダッシュボード KPI と月次請求画面の冒頭で同じ数字が並ぶ二重表示は不要

### 1-3. 「よく使う操作」が単なるショートカット
- サイドバーの項目を二重に並べているだけ
- 業務上意味のあるショートカット（「本日の発注を確定する」「鈴木さんの入院ステータス変更」）になっていない

### 1-4. 業務ロジックの不可視化
- アレルギー・禁忌・食事形態が利用者詳細の文字列で埋もれている（食事配膳の事故リスク）
- 業者別の締切時刻が表示されていない（締切直前の判断ができない）
- 「請求漏れ」「発注漏れ」「在庫不足」のアラート系がダッシュボードに存在しない
- 食数の「前日からの差分」が見えない（突発的な変動の発見が遅れる）

### 1-5. アクションのファネルが不在
- ダッシュボードで気付く → 詳細を確認 → 修正する → 確定する、までの導線が画面間で完結していない
- 詳細を見るために別画面に遷移し、戻ると状態が失われる

## 2. 設計の核となる原則

### 2-1. 「気付き → 判断 → アクション → 確定」を 1 動線で完結させる
ダッシュボードで気付いた問題を、画面遷移せずに **サイドドロワー** で内訳確認・修正・確定までできるようにする。

### 2-2. 数字の隣に必ず「内訳リンク」「最終確定日時」「次のアクション」を置く
裸の数字を出さない。`本日昼A社 14` の隣に `内訳 > / 前日 +2 / 締切 10:30（あと 1h 50m）` を付ける。

### 2-3. ステータスは「色 + テキスト + アイコン」の3点セット
色盲・印刷時・モニタ違いに耐える設計。

### 2-4. カード乱用禁止、テーブル + ドロワー優先
- カードはダッシュボード上段の概観・グラフ専用
- 一覧はテーブルで密度を維持
- 詳細・編集はサイドドロワー（左右どちらかから出てくるパネル）で画面遷移を最小化

### 2-5. 業務語彙に揃える
- 「タスク」「アラート」「お知らせ」を明確に分ける（後述の情報レーン参照）
- 「キャンセル」は「定期キャンセル」「単発キャンセル」「ステータス連動停止」を区別
- 「確定」は「発注確定」「請求確定」を区別
- 「未確定理由」を表示する（請求漏れと未確定は別）

### 2-6. ミス防止モーダルを「破壊的操作のみ」に絞る
- 発注確定、請求確定、確定解除、ステータス変更（食事自動停止が走るもの）にのみ確認モーダル
- 通常編集・追加では確認モーダルを出さない（疲労を増やすため）

## 3. 情報モデルの再定義（4つの情報レーン）

ダッシュボードと通知系を、以下の **4つのレーン** に分離する。

| レーン | 定義 | 例 | 押下時の挙動 | 通知音/重要度 |
|---|---|---|---|---|
| **① アラート** | システムが検知した「異常」。放置すると業務事故になる | 在庫が最低在庫以下、本日の発注未確定（締切〜30分前）、書類期限切れ、請求漏れ検知、食事設定がないのに食事が発注されている | 該当画面の **修正ドロワー** を開く | 高 |
| **② タスク** | 担当者が「動く」必要があるアクション | ケアマネ面談調整、退去精算、請求書送付、家族連絡、棚卸 | タスク詳細ドロワー（編集・完了マーク） | 中 |
| **③ お知らせ** | 施設管理者から職員へのアナウンス | 「5/15 から B 社昼食締切が 10:30 に変更」「6月の連休配達日程」 | お知らせ詳細ページ | 低 |
| **④ アクティビティログ** | 何が起きたかの記録（既読化のみ） | 「鈴木様 入院中に変更（5/10 17:20）」「日用品在庫補充：田中 太郎 が +20」 | 既読化 + 元画面へのリンク | 情報のみ |

ダッシュボードでは **① → ② → ③ → ④ の順** に出す。④ は最下部にコンパクトに。

### 3-1. アラートの種別マスタ
以下を仕込み、それぞれに「検知条件」「重要度」「責任ロール」「次のアクション」を持たせる。

| アラート | 検知条件 | 重要度 | 責任ロール | 次のアクション |
|---|---|---|---|---|
| 発注未確定（緊急） | 締切まで 60 分以内、未確定 | 高 | 事務 | 業者別発注表へ → 確定 |
| 発注未確定（注意） | 締切まで 4 時間以内、未確定 | 中 | 事務 | 業者別発注表へ |
| 在庫不足 | 在庫 < 最低在庫 | 中 | 事務 | 発注候補表へ |
| 在庫切迫 | 在庫 < 最低在庫 × 0.5 | 高 | 事務 | 発注候補表へ |
| 書類期限切れ | 期限を過ぎている | 高 | 事務 | 書類管理へ |
| 書類期限間近 | 30 日以内 | 中 | 事務 | 書類管理へ |
| 請求漏れ検知 | 食事は出ているが、食費請求が 0 円 | 高 | 事務 | 利用者請求明細へ |
| 請求漏れ検知 | 月末を過ぎた前月分が未確定 | 高 | 事務 | 月次請求へ |
| 食事設定なし | 入居中なのに食事設定が登録されていない | 中 | 事務 | 利用者詳細「食事」タブへ |
| 自動停止解除漏れ | 退院済みなのに食事停止中 | 高 | 事務 | 利用者ステータスドロワーへ |
| 単価未設定 | 食費単価マスタが空、または当月の単価がない | 高 | 管理者 | マスタへ |

- **左サイドバー**：トップ階層のみ（最大8項目）。ネストは出さない

- **戻る動線**：ブラウザの戻るが常に意味のある画面に戻る（クエリパラメータでフィルタ状態を保持）

## 5. サイドバー再設計

### 5-1. 方針
- 幅 200px → **210px** に微増、ただし密度を上げる
- グルーピングを導入：**運用 / 請求 / 在庫 / 管理**
- 各項目に **アイコン + テキスト + 件数バッジ**
- 「アラート」と「タスク」をサイドバー上部に常設（件数バッジが赤・橙で常時見える）

### 5-3. 細部仕様
- アクティブ項目は左に 3px の縦線（ブランド色）、背景 `brand-50`、文字 `brand-700`、太字

- 件数バッジは円形、最大 99（それ以上は `99+`）
- アラートのバッジは赤 (`err-600`)、タスクは橙 (`warn-600`)、それ以外は灰 (`ink-500`)

## 6. ダッシュボード再設計

### 6-1. 設計意図
**今日この施設で起きていることが 5 秒で分かり、次のアクションへの導線が明確**にする。
ダッシュボードは「眺める画面」ではなく「アクションの起点」。

### 6-2. レイアウト（上から）

```
┌─ 1. インシデント・バー（最上部固定）─────────────────────┐
│ ⚠ 緊急 2 件：本日昼食 A 社 締切まで 1h 50m（未確定 3 名）│
│ ⚠ 鈴木様 退院済みなのに食事停止中（要解除）              │
└──────────────────────────────────────────────────────┘

┌─ 2. 今日の食事発注ステータス ────────────────────────┐
│ 朝 ✓確定済    昼 ⚠未確定（3名）     夕 待機中        │
│ [本日の発注詳細へ ＞]                                │
└──────────────────────────────────────────────────────┘

┌─ 3. 入居・財務 KPI ──────────────────────────────────┐
│  入居 10  入院 1  外泊 1  一時帰宅 1  空室 3        │
│  今月請求予定 ¥6,432,100   未確定 10件             │
└──────────────────────────────────────────────────────┘

┌─ 4. 影響イベント（食事/請求に影響するもの）──────────┐
│ ・鈴木様：入院（5/5〜）食事停止中　[ステータス変更]   │
│ ・田中様：外泊（5/10〜5/12）食事自動停止             │
│ ・加藤様：一時帰宅（5/9〜5/11）食事自動停止           │
│ ・新規入居予定：渡辺様（5/20予定）                   │
└──────────────────────────────────────────────────────┘

┌─ 5. アラート（要対応・3件）────────┬─ 6. タスク（5件）──┐
│ ⚠ 在庫不足：おむつ L (24/40)        │ ◆ 5/12 病院連絡   │
│ ⚠ 書類期限間近：鈴木様 介保証 (5/15)│ ◆ 5/15 ケアマネ   │
│ ⚠ 請求漏れ疑い：高橋様 食費 0円    │ ◆ 5/20 受給者証   │
└────────────────────────────────────┴────────────────────┘

┌─ 7. 申し送り（タイムライン）─────────────────────────┐
│ 09:40  田中：鈴木様 退院日が 5/13 に確定              │
│ 09:15  鈴木：山本様 木曜キャンセルですが今週は出席    │
└──────────────────────────────────────────────────────┘

┌─ 8. アクティビティログ（折り畳み・既読）─────────────┐
│ ▶ 12 件 / 過去 24 時間 [展開]                        │
└──────────────────────────────────────────────────────┘
```

### 6-3. 各セクションの仕様

#### 6-3-1. インシデントバー（最上部）
- 高さ 44px、`err-50` 背景 + `err-600` の左縦線
- 最大 3 件、4 件目以降は「他 N 件 > 」リンク
- 該当画面のドロワーを開く（画面遷移しない）

#### 6-3-2. 食事発注ステータス
- 朝・昼・夕の3つのステータスチップを横並び
- 各チップに「✓確定済 (確定日時)」「⚠未確定 (締切まで N 時間)」「待機中 (締切まだ)」
- チップクリックで日別詳細ドロワー

#### 6-3-3. 入居・財務 KPI
- 横並び5項目 + 財務2項目 = 計7
- 数字をクリックで対応する一覧画面（フィルタ済み）

#### 6-3-4. 影響イベント
- 食数・請求に影響する利用者ステータスのまとめ
- 過去 30 日 + 未来 30 日を表示
- 各行に「ステータス変更ドロワー」リンク

#### 6-3-5. アラート・タスク（2カラム）
- 左：アラート（自動検知、急ぎ順）
- 右：タスク（人手、期限順）

#### 6-3-6. 申し送り
- タイムライン形式、過去 24 時間
- ＠メンション対応（Phase 2）

#### 6-3-7. アクティビティログ
- 初期は折り畳み、展開で過去 24 時間の全件

## 7. タスク／アラート／お知らせ／アクティビティ（4つのレーン詳細）

### 7-1. アラート（/inbox/alerts）
**画面**：自動検知された異常の一覧

| 列 | 内容 |
|---|---|
| 種別アイコン | 在庫 / 書類 / 発注 / 請求 / ステータス |
| 重要度 | 高 / 中 / 低 |
| 発生日時 | 検知された日時 |
| 内容 | 「在庫不足：おむつ L（残 24／最低 40）」 |
| 対象 | 利用者 or 商品 or 業者 |
| 影響範囲 | 「本日昼食 3名分の発注に影響」 |
| 担当ロール | 事務 / 管理者 |
| 次のアクション | ボタン：[修正ドロワーを開く]、[該当画面へ]、[解決済として保留] |
| ステータス | 未対応 / 対応中 / 解決 / 抑制 |

操作：
- 解決すると自動的に既読
- 「抑制」はそのアラート種別を 24 時間ミュート（理由必須）

### 7-2. タスク（/inbox/tasks）
**画面**：人手の作業項目

| 列 | 内容 |
|---|---|
| ステータス | 未対応 / 対応中 / 完了 / 見送り |
| タスク名 | 「ケアマネ面談調整」 |
| 区分 | 請求 / ケア / 書類 / 連絡 / 棚卸 / その他 |
| 対象利用者 | 利用者名（任意） |
| 担当 | 職員名 |
| 期限 | YYYY-MM-DD |
| 優先度 | 高 / 中 / 低 |
| 次のアクション | [完了]、[編集]、[コメント追加] |

機能：
- 繰り返しタスク（月次の請求書送付など、Phase 2）
- 標準テンプレ（入居時セット、退去時セット）

### 7-3. お知らせ（/inbox/announcements）
**画面**：管理者から職員への一方向アナウンス

| 列 | 内容 |
|---|---|
| タイトル | 「5/15 から B 社昼食締切変更」 |
| 本文 | 詳細 |
| 投稿者 | 管理者名 |
| 投稿日 | 日時 |
| 既読/未読 | 自分の状態 |
| 期間 | 「掲示終了：6/30」など |
| 重要 | 「ピン留め」フラグ |

機能：
- 既読率を投稿者が確認できる
- 配信対象ロール（全員 / 事務のみ / 現場のみ）

### 7-4. アクティビティ（/inbox/activity）

|---|---|

### 19-3. アラート検知バッチ
- 毎日 6:00 と 13:00 にバッチ実行
- 全アラート種別の検知ロジックを走らせ、`alerts` テーブルに INSERT
- 解決済アラートと同条件のものは重複作成しない
- リアルタイム検知が必要な項目（請求確定時の請求漏れ）はその場で実行

### 19-4. 食事発注ロジック（既存ロジックの拡張）
v1 の集計ロジックに加えて以下を実装：
- **前日比**：当日値 − 前日の確定値（前日未確定なら null）
- **業者別締切までの残時間**：締切時刻 − 現在時刻、リアルタイム表示
- **未確定者リスト**：その業者・その食事区分の対象者で、当日の操作（追加/削除/手動調整）が反映されていない利用者

### 19-5. 請求漏れ検知ロジック
- 食事カレンダーで「対象者リストに含まれている」のに、月次請求明細に該当食費がない
- 利用者は入居中なのに、固定費が未設定
- 月末を過ぎているのに、前月の請求が未確定

### 19-6. ステータス変更時の自動処理
- 入院/外泊/一時帰宅にする → 期間中の未確定食事は自動キャンセル、確定済食事は **触らない**
- 退院/外泊終了 → 期間後の食事を自動復帰、確定済は触らない
- 退去 → 退去日以降の食事を自動停止、固定費を退去月設定に従って調整
- すべて `user_status_changes` に履歴を残し、影響件数を `audit_logs` に記録

### 19-7. 食札 PDF 生成
- React-PDF または Puppeteer
- 利用者ごと、食事区分ごとに 1 ラベル
- A4 1 枚に複数ラベル、A6 サイズの食札としてカット可能なレイアウト
- 必ず：氏名、部屋、食事区分、形態、**アレルギー（赤字）**、禁忌、特記

### 19-8. テスト戦略
- ロジック層（meal_orders 集計、請求漏れ検知、ステータス連動キャンセル）に **必ず単体テスト**
- E2E は MVP リリース前に主要動線 5 本（入居→食事設定→発注→キャンセル→請求確定）
- アクセシビリティ：キーボード操作、コントラスト比、スクリーンリーダー

## 20. まとめ

本提案書は、v1 プロトタイプの「画面はある」状態から、**商用 SaaS として現場のミスを減らし業務を回せるプロダクト** への引き上げを目的に作成した。

最重要の変更：
1. 情報を **4 つのレーン**（アラート / タスク / お知らせ / アクティビティ）に分けた
2. **アラート機構** を追加し、業務事故になりうる異常を自動検知する
3. **食事日別詳細画面** を新設し、業者別 + 締切時間 + 確定ボタンの動線を 1 画面で完結させた
4. 利用者詳細の **アレルギー・禁忌を最上部に強調表示** し、配膳事故を防ぐ
5. **請求漏れ検知** をダッシュボードと請求画面に組み込み、月末の発見漏れを防ぐ
6. **ステータス変更ドロワー** で、入院・外泊が食事・請求に与える影響を変更前に確認できる

これらの設計により、B-CareHub は「綺麗な画面」ではなく「業務事故を減らす実務プロダクト」になる。
